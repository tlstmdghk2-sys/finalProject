Core Implementation · Security Critical
Prepare → Request → Verify: 서버 측 금액 검증을 통한 위변조 방지
💡 결제 금액을 클라이언트만 믿으면 안 됩니다. 공격자가 브라우저 개발자 도구에서 결제 금액을 조작하면 실제 결제 금액보다 낮은 금액으로 결제가 완료될 수 있습니다. 서버가 merchantUid와 금액을 사전 등록하고(prepare), 결제 완료 후 서버에서 재검증(verify)하는 3단계 구조로 이를 원천 차단합니다.

알라딘 API 데이터 파이프라인

Data Engineering · Python + OracleDB
Aladin OpenAPI → ISBN UPSERT → Oracle DB 적재 파이프라인
💡 외부 API 데이터를 그냥 INSERT하면 중복 실행 시 데이터가 쌓입니다. ISBN이라는 도서의 고유 식별자를 기준으로 이미 존재하면 UPDATE, 없으면 INSERT하는 UPSERT 패턴을 적용해 멱등성 있는 파이프라인을 구축했습니다.

row = cursor.fetchone()

if row:
    # 기존 BOOK 업데이트: IMG_URL만 갱신
    img_url = book.get("cover")
    cursor.execute("""
        UPDATE BOOKS
        SET IMG_URL = :1, UPDATED_AT = SYSDATE
        WHERE ISBN = :2
    """, (img_url, isbn))
else:
    # 새 도서 INSERT
    desc = (book.get("description") or "")[:1000]
    cursor.execute("""
        INSERT INTO BOOKS (
            BOOK_ID, TITLE, AUTHOR_ID, PUBLISHER_ID,
            DESCRIPTION, ISBN, PRICE, IMG_URL,
            CREATED_AT, UPDATED_AT
        ) VALUES (
            SEQ_BOOKS.NEXTVAL, :1, :2, :3,
            :4, :5, :6, :7, SYSDATE, SYSDATE
        )
    """, (title, author_id, pub_id,
              desc, isbn, price, img_url))

    if not name or name.strip() == "":

시스템 안정성을 위한 문제 해결

💳 PortOne 결제 멱등성 문제
Problem
새로고침 또는 네트워크 지연 시 동일한 결제 요청이 중복으로 처리되는 문제 발생
Cause
merchantUid 중복 검증 로직 부재로 인해 동일 결제 요청이 여러 번 처리될 수 있었음
Solution
merchantUid를 기준으로 UNIQUE 제약 키 적용으로 중복 결제 원천 차단 → 단 한 번만 결제가 승인되도록 로직 개선
📚 Aladin API 데이터 적재 오류
Problem
API 응답 데이터 삽입 시 ORA-12899: value too large 오류 발생
Cause
카테고리명과 도서 설명(description) 데이터가 VARCHAR2 컬럼 한도를 초과하여 삽입 불가
Solution
DBA 협의 후 해당 컬럼을 CLOB 타입으로 변경 → DB 구조 개선으로 안정적인 데이터 적재 환경 구축
